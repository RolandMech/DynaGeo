Wie wird DynaGeoJ in eine HTML-Seite eingebunden?

Normalerweise werden Sie den in diesem Abschnitt beschriebenen HTML-Code nicht selbst erzeugen müssen. Dies erledigt EUKLID DynaGeo für Sie, wenn Sie den Befehl Datei | Als DynaGeoJ-HTML-Seite speichern... anwählen. Falls Sie das HTML-Dokument jedoch weiterbearbeiten wollen oder den erzeugten HTML-Code in anderen HTML-Dokumenten weiterverwenden wollen, sollten Sie verstehen, wie DynaGeoJ eingebunden wird. Daher wird hier der HTML-Code für die Verwendung von DynaGeoJ beschrieben:

Sie haben mit EUKLID DynaGeo eine Zeichung erstellt und diese in einer GEO-Datei abgespeichert, z.B. unter C:\Programme\DynaGeo\Daten\BEISPIEL.GEO. Wenn Sie nun diese Zeichnung in einer HTML-Seite angezeigt haben wollen, dann müssen Sie an der entsprechenden Stelle des HTML-Quelltextes die folgende "<APPLET>...</APPLET>"-Deklaration einfügen:

    <APPLET

      code="applet.DynaGeoJ"
      archive="../dynageoj.jar"
      width=501
      height=330
    >
    <param name="geofile" value="beispiel.geo">
    </APPLET>

Ein paar Erläuterungen zu den einzelnen Parametern:

  • "code" enthält eine eindeutige Identifikation für das DynaGeoJ-Applet.
      Frühere Versionen von DynaGeoJ hatten hier einen unschönen Pfad-Bandwurm stehen. Dieses Manko ist nun beseitigt, ehe die Versionsnummer 1.0 erreicht wird ;-)

  • "archive" enthält einen Pfad zur JAR-Datei, die das DynaGeoJ-Applet enthält: entweder den relativen Pfad von der HTML-Datei selbst zur JAR-Datei (wenn sich beide Dateien auf demselben Server befinden) - oder eine volle Internet-Adresse, unter der der DynaGeoJ-Viewer verfügbar ist (z.B. "http://www.dynageo.de/download/dynageoj.jar").

  • "width" und "height" geben die Abmessungen des Ausgabefensters (in Pixeln) an.

  • "param" enthält den Namen der GEO-Datei, die die Zeichnung enthält:
    "name" muss hier den Wert "geofile" haben, und in "value" steht der Name (ohne Pfad!) der anzuzeigenden GEO-Datei. Dabei wird vorausgesetzt, dass sich diese Datei im selben Verzeichnis befindet wie die HTML-Datei.



Weitere Hinweise:

  • Wenn Sie das HTML-Dokument weiterbearbeiten, müssen Sie ein paar Syntax-Regeln beachten. Damit Sie sich die Möglichkeit einer zukünftigen Überarbeitung der Datei in EUKLID DynaGeo erhalten, muss sichergestellt werden, dass EUKLID DynaGeo die DynaGeoJ-Objekte auch stets zuverlässig erkennen kann:
    • Groß- und Kleinschreibung spielt bei den Tag-Namen keine Rolle. Seien Sie aber bei Dateinamen in dieser Beziehung sehr vorsichtig, wenn Sie ihre Dateien ins Netz stellen wollen: im Gegensatz zur Praxis unter WINDOWS ist für LINUX-Server die Groß- und Kleinschreibung bei Pfadangaben signifikant!
    • Achten Sie stets darauf, dass die umschließenden Tags "<APPLET..." und "</APPLET>" immer alleine in einer eigenen Zeile des HTML-Quelltextes stehen. Es darf also weder vor dem "<APPLET..."-Tag noch nach dem "</APPLET>"-Tag irgend eine andere wesentliche HTML-Information stehen. Lediglich ein Zeilenumbruch ("<BR>") ist erlaubt.
    Am besten belassen Sie die "<APPLET>"-Tags einfach in dem Zustand, in dem sie von EUKLID DynaGeo erzeugt wurden!

  • Bei neu erzeugten DynaGeoJ-HTML-Seiten stimmen üblicherweise der Dateiname des HTML-Dokuments und der Dateiname der referenzierten Zeichnung bis auf die Erweiterung überein. Sie können diese strenge Synchronisation zwischen den Dateinamen aufbrechen, indem Sie im Fenster "Als DynaGeoJ-HTML-Seite speichern" eine andere, eventuell schon bestehende HTML-Datei als Ziel auswählen. Auf diese Art und Weise ist es sogar möglich, mehrere DynaGeoJ-Fenster in eine HTML-Seite einzufügen. Beachten Sie aber, dass Sie in einem solchen Fall selbst dafür verantwortlich sind, dass beim Transport der HTML-Datei in eine andere Umgebung auch alle zugehörigen GEO-Dateien mitgenommen werden! Normalerweise sollten Sie sich auf ein DynaGeoJ-Objekt pro HTML-Seite beschränken und die Dateinamen synchron belassen.

  • Wenn Sie bei selbsterstellten DynaGeoJ-HTML-Dokumenten für "archive" eine vollständige Internet-Adresse verwenden, dann erhalten Sie frei verschiebbare HTML-Dokumente; hingegen ist ein relativer Pfad zu einer lokalen JAR-Datei natürlich nicht mehr gültig, wenn Sie das Dokument auf einen anderen Rechner transportieren!
    Trotzdem empfiehlt es sich, für eigene Veröffentlichungen im Netz einen relativen Pfad zu verwenden, der vom HTML-Dokument zu einer lokalen Kopie von "dynageoj.jar" auf dem eigenen Server zeigt. Dies sichert Ihnen die Unabhängigkeit Ihrer Site von externen Ressourcen!
